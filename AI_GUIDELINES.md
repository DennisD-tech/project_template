# AI 协作准则

本文档定义 AI 在本项目中的默认工作方式，用于**提高交付质量**、**降低沟通成本**、并让每个新项目都能快速进入稳定迭代节奏。

## 核心目标

- **交付稳定可靠的产品**：正确性优先，其次是可维护性与迭代效率。
- **默认追求**：
  - 代码效率（在可读与可维护前提下）
  - 架构稳定（清晰边界、可扩展、可替换）
  - 可维护可复用（减少偶发复杂度、避免“只懂作者”的实现）

## 合作方式与沟通

- **直说问题**：当我的要求与“稳定可靠、可维护”冲突时，请直接指出风险与后果，并给出替代方案。
- **主动做取舍分析**：遇到多方案可行时，请比较 trade-off（复杂度、成本、性能、安全、未来演进），并给出推荐与理由。
- **避免过度防御**：不要为了“理论兜底”引入沉重架构；拒绝马奇诺式预防工程。
- **约定大于配置**：优先遵循既有约定与项目风格；用代码表达事实，文档只记录必要决策。

## 语言与表达

- **默认中文**：AI 生成/修改的文档、提交信息、沟通内容默认使用中文（不强制）。
- **专有词汇不硬翻**：领域术语按行业常用表达（例如 API、trade-off、runtime、Edge/Node）。
- **命名以代码约定为准**：若项目对命名、目录结构、接口命名有独立规则，请以项目内规范文件为最高优先级。

## 文档规范

### 原则

- **只保留必要文档**：文档要可被持续维护，否则宁缺毋滥。
- **内容要可执行**：写“结论、约束、决策依据”，少写流水账。
- **及时更新**：改动影响使用方式/约定时，同步更新相关文档。
- **清理临时文件**：阶段性完成后，及时清理或归档临时文档。
- **根目录 vs `docs/`（分工）**：
  - **根目录**：放**高频、可执行**的内容（节奏、验证命令、关键决策速记），便于人与 agent 每天用、`@` 引用方便。
  - **`docs/`**：放**需沉淀、偏正式或偏长文**的内容（需求背景、架构、质量体系、运维），默认由人主导维护；agent **仅在任务需要时**阅读对应章节。

### 根目录（模板默认）

- `README.md`：项目简介、安装与启动、常用命令；至少用一行指向 `docs/00-overview/GETTING-STARTED.md`（若已建立）以及部署入口 `docs/07-operations/`（若已建立）。
- `PLAN.md`：大尺度计划与里程碑，最小颗粒度为模块/里程碑级别。
  - 可用特殊标记（例如 `---->`）标识整体推进状态。
  - 拟执行项应拆分/下放到 `TODO.md` 与 `WIP.md`。
  - 越靠近执行层越短、越具体；越靠近计划层越稳定、越抽象。
- `TODO.md`：中长期**仓库内**任务池（由 `PLAN.md` 拆分）。粒度介于 `PLAN.md` 与 `WIP.md` 之间。
  - **与 GitHub Issues 等区分**：本文件是“仓库内计划”；若团队以 Issues 为主，可在此文件中只做索引与链接，避免双份权威。
- `WIP.md`：当前迭代/当前会话的**下一步可执行**清单（通常由 `TODO.md` 拆分）。
- `DEV_NOTE.md`：长期有效的踩坑记录、基建知识、**关键决策**（决策依据 + 最终结论）；不要求流水账。
- `TESTING.md`：**一页纸跑通测试**（安装前提、标准测试命令、测试目录/命名约定、与 CI 是否一致）；详述与用例库放在 `docs/06-quality/`，此处只保留链接与摘要。

### `docs/` 目录结构（模板默认）

正式文档集中在 `docs/`，按主题分目录（无内容的目录可留空或减少子文件，但**不要随意改名编号**以便全团队对齐）。

```text
docs/
├── 00-overview/           # 总览：索引、背景、分工；新人上手；版本/变更索引（可选）
├── 01-requirement/        # 需求：PRD、用户故事、需求变更
├── 02-research/           # 调研：技术调研、竞品、可行性、方案对比
├── 03-architecture/       # 架构：模块设计、数据模型/字典、接口契约（如 OpenAPI）
├── 04-design/             # 设计：原型、交互说明、设计稿链接（无则仅保留说明）
├── 05-engineering/        # 工程：环境/构建/CI 说明摘要、编码与目录约定（可与仓库内规范互链）
├── 06-quality/            # 质量：测试策略、用例与报告、缺陷汇总（与根目录 TESTING.md 配合）
├── 07-operations/         # 运维：上线、回滚、故障处理；默认不要求 agent 频繁阅读
└── _templates/            # 可选：PRD、评审纪要等模板，降低撰写门槛
```

- **会议纪要**：不单独占顶层；可放在 `docs/00-overview/meetings/`，或需求相关纪要放在 `docs/01-requirement/meetings/`。
- **需求/调研/选型主文**：分别落在 `01-requirement/`、`02-research/`；技术选型说明可放在 `02-research/` 或与 `03-architecture/` 互链，避免根目录再堆 `PRD.md` / `TRD.md`（除非你刻意保留一份根目录拷贝，则须注明与 `docs/` 哪一份为权威）。

### 我想快速找信息（新人/按需）

| 目的 | 建议先读 |
|------|-----------|
| 把项目跑起来 | `README.md` → `docs/00-overview/GETTING-STARTED.md`（若存在） |
| 当前要做什么、从哪条任务开始 | `WIP.md` → `TODO.md` → `PLAN.md` |
| 跑测试、补单测 | `TESTING.md` → `docs/06-quality/`（策略与详细用例） |
| 需求范围与验收 | `docs/01-requirement/` |
| 架构与接口契约 | `docs/03-architecture/` |
| 部署、回滚、故障 | `docs/07-operations/`（日常开发任务通常不必读） |
| 关键决策与踩坑 | `DEV_NOTE.md` |

## 开发流程

- **接到任务先做最小计划**：明确目标、边界、验收标准；必要时拆解任务与风险点。
- **优先可验证**：能写测试就写测试；无法自动化时，明确可重复的验证步骤。
- **逐步实现与验证**：按拆解逐项完成，保持每一步可运行/可验证。
- **需要时沉淀决策**：将“关键约束/关键取舍/关键风险”写入 `DEV_NOTE.md`；若直接影响架构或接口契约，同步更新 `docs/03-architecture/` 中对应说明。
- **提交前常规检查（按项目实际命令调整）**：
  - 格式化：例如 `pnpm run format`（若使用 Biome/Prettier）
  - 类型检查：例如 `pnpm run typecheck`
  - 构建：例如 `pnpm run build`

### Bug 修复准则

- **先稳定复现**：构造数据或操作路径，使 bug 可重复出现。
- **把复现固化**：尽量写成测试用例或脚本化验证步骤。
- **修复后防回归**：确保用例稳定通过，避免同类问题在后续迭代反复出现。

### 工作习惯

- **小步快跑**：每完成一个清晰的小目标（一个功能点/一个 API/一个 bugfix）就提交一次。
- **允许事后整理**：在任务完全验收后，如需要可做提交整理（例如 squash/rebase），但不要牺牲过程可追踪性。
- **做完再推进**：完成任务后进行必要回归、清理临时文档、识别可选优化点，再开始下一件事。

### 最小单元流程

以“文件”为中心的最小交付闭环，默认按以下顺序执行（除非任务另有约束）：

- **从临时文件读取下一步**：
  - 优先从 `WIP.md` 读取“下一条可执行任务”
  - 若 `WIP.md` 没有可执行项，再从 `TODO.md` 拆分出本轮 `WIP.md`
  - 若 `TODO.md` 也不足以落地，再回到 `PLAN.md` 补齐/澄清范围与里程碑
- **创造/修改代码文件**：实现任务所需的最小改动，保持边界清晰、避免顺手“扩需求”。
- **同步修改/新增单元测试**：用测试固化预期与边界，优先覆盖本次变更引入的关键行为与回归点。
- **运行并通过测试**：以根目录 `TESTING.md` 中的命令为准；确保可重复通过。若无法自动化，写下可重复的验证步骤并执行验证。
- **回顾并更新临时文件**：
  - 将已完成的事项从 `WIP.md` 删除（或标记为完成并清理）
  - 若 `WIP.md` 被清空，检查是否需要逐层回收与清理：`WIP.md` → `TODO.md` → `PLAN.md`
  - 若在实现过程中产生了**关键决策**（取舍、约束、架构边界、不可逆选择等），将“决策依据 + 最终结论”写入 `DEV_NOTE.md`（必要时在 `PLAN.md` 做一致性同步）；若涉及架构或接口契约，同步更新 `docs/03-architecture/` 中对应说明。

## 代码规范（默认约定，可被项目覆写）

- **默认 TypeScript**：尽可能写清类型；避免 `any`（除非明确有理由并隔离影响）。
- **少靠 JSDoc**：优先用 TypeScript 类型系统表达约束。
- **命名约定**：
  - 变量/函数：`camelCase`
  - 类/类型/接口：`PascalCase`
  - 常量：`UPPER_SNAKE_CASE`
  - 文件/目录：`kebab-case`
  - 避免不必要缩写；布尔值使用 `is/has/can` 前缀
  - 函数用动词或动宾；类用名词
- **组件与文件体量**：单文件尽量控制在 300 行附近，超过 400 行应优先拆分（以可读性与边界清晰为目标）。
- **图标命名避免歧义**：例如使用 `SaveIcon` 而不是 `Save`（若项目图标库存在同名冲突风险）。
- **函数声明风格（如无项目约定冲突）**：优先 `function foo(){}`，避免无必要的 `const foo = () => {}`（便于栈追踪、提升可读性）。
- **SVG 使用策略**：避免内嵌 SVG；优先使用统一的图标库（例如 `lucide`），以保持一致性与可维护性。

## 环境与工具（模板默认，可按项目调整）

- Node.js：一般默认最新 LTS（项目可指定，如 `>=24`）
- python: 一般默认最新版本, 如python3.13
- 包管理：默认 `pnpm`（若项目另有约束，再按约束调整）
- 构建：例如 `vite`（若本项目采用）
- 测试：例如 `vitest`（若本项目采用）

## 安全与敏感信息

- **默认不读取敏感文件**：例如 `.env`、密钥、凭据、生产配置等。
- **需要敏感数据时的做法**：由 AI 编写脚本/命令与说明，你来本地运行并仅回传必要的、已脱敏的输出。
